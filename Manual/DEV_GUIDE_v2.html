Como o dashboard calcula o status de cada nível
💡
O status de um Marco, Grupo ou Eixo é calculado automaticamente com base nas suas tarefas — sempre usando o pior status encontrado.
📊 Eixo herda o pior status dos seus Grupos
📁 Grupo herda o pior status dos seus Marcos
🎯 Marco herda o pior status das suas Tarefas
Exemplo: se um grupo tem 10 tarefas "Feitas" e 1 "Atrasada" → o grupo aparece como Atrasado.

Regras de negócio do código
Lógica implementada em index.html (arquivo único, responsivo). Referência para quem vai manter ou evoluir o dashboard. Última revisão: Jul/2026
Grupo 1 — Hierarquia de status (propagação bottom-up)
RN-01 Ranking de severidade
Ordem de prioridade usada em todas as comparações de "pior status" (pior → melhor):
Atrasado › Risco de atraso › Em andamento › A iniciar = (vazio) › Definir datas › Feito › Cancelado/Congelado = Cancelado
RN-02 Status do marco = pior status entre as subtasks
Após preprocessStatuses, o status do marco é sobrescrito pelo pior status das suas tarefas. Se não tem subtasks, o marco é tratado como folha diretamente.
RN-02a Exceção: marco só vira "Feito" se 100% das subtasks são Feito/Cancelado
Mesmo que o marco esteja marcado como "Feito" na planilha, o código valida se todas as subtasks também estão Feito ou Cancelado — se não estiverem, usa o pior status delas.
RN-03 Status do grupo = pior status entre os marcos
RN-04 Status do eixo = pior status entre os grupos
RN-13 Datas de marco e grupo — propagação bottom-up
As datas exibidas para marcos (com subtasks) e grupos são calculadas a partir das tarefas filhas, não lidas diretamente da planilha:
ElementoData inícioData fim
Marco (com subtasks)mínimo das datas de início das tarefasmáximo das datas de fim das tarefas
Grupomínimo das datas de início dos marcosmáximo das datas de fim dos marcos
⚠️ Marcos sem subtasks continuam com as datas que vêm direto da planilha.
🛡️
Imunidade: Marcos, grupos e eixos com status Cancelado/Congelado ou Cancelado não são recalculados — ficam como estão, independente das subtasks.
Grupo 2 — Auto-correção por data (preprocessStatuses)
Roda sobre cada tarefa/subtask individual, sobrescrevendo o status da planilha com base nas datas:
Status na planilhaCondição de dataStatus resultante
A iniciarData de início já passouRisco de atraso
A iniciarSem data de iníciomantém A iniciar
Qualquer outro ¹Data de fim < hojeAtrasado
Qualquer outro ¹Data de fim ≤ hoje + 7 diasRisco de atraso
Qualquer outro ¹Data de fim okmantém como está
¹
Exceto Feito, Cancelado/Congelado e Cancelado — esses nunca são sobrescritos por data.
Grupo 3 — KPIs
RN-05 KPI "Marcos Concluídos" — desktop
Marco conta como concluído se: status === 'Feito' E todas as subtasks são Feito/Cancelado. Marcos com status Cancelado/Congelado são excluídos do total.
RN-06 KPIs de tarefas — desktop (base de cálculo)
Tarefas ativas = todas exceto Cancelado e Cancelado/Congelado. Marcos sem subtasks entram diretamente como tarefas na base de cálculo.
RN-07 KPI hero mobile = % de grupos com status "Feito"
Fórmula: feito / totalGrupos * 100 — conta grupos, não marcos nem tarefas.
Grupo 4 — Estrutura dos dados (parsing)
RN-08 Detecção dinâmica da semana atual
Busca da direita para a esquerda no header do sheet a coluna SEMANA ATUAL (normalizado sem acentos). Nas próximas 5 colunas localiza PROGRESSO e ENCAMINHAMENTO. Se não achar, usa fallback hardcoded nas colunas 37 e 39.
RN-09 Montagem da hierarquia EIXO → GRUPO → MARCO → TAREFA
• Linha onde tarefa === grupo → summary row do grupo
• Linha onde tarefa === marco → o próprio marco
• Demais linhas com mesmo marco → subtasks do último marco visto
RN-10 Filtro de linhas inválidas em REQS
Ignora linhas onde col[0] é: vazio, um dos headers de agrupamento (Em andamento, Cotação, etc.), ou número decimal (1.0 → converte para 1).
Grupo 5 — Filtros de exibição
RN-11 Modo marco vs. tarefa — filterLevel (desktop)
• marco: o filtro de status age sobre o status calculado do marco inteiro
• tarefa: o filtro age sobre as subtasks dentro do marco expandido
RN-12 Filtro de status mobile — age sobre grupos
O clique no strip KPI filtra a lista de eixos/grupos pelo status do grupo.
⚠️
Qualquer alteração na lógica de status ou KPIs deve ser verificada nas duas cópias de dados usadas pelo dashboard — o cronograma principal (WBS) e o da aba Diretoria (WBS_DIR) — pois usam funções de pré-processamento diferentes (preprocessStatuses vs preprocessStatusesDiretoria).
Grupo 6 — Aba Diretoria
RN-14 WBS_DIR é cópia congelada no load, não referência viva
WBS_DIR.length=0; JSON.parse(JSON.stringify(newWBS)).forEach(e=>WBS_DIR.push(e)); — é um deep copy tirado no momento da carga dos dados. A partir daí, WBS e WBS_DIR vivem desacoplados: editar o status manualmente numa árvore não propaga para a outra. Isso é intencional (permite status divergente para o Comparativo), mas quebra a expectativa de "é a mesma árvore".
RN-15 Comparativo casa tarefas por eixo/grupo/marco — não por ID
buildComparativoModal() encontra divergências de status comparando WBS × WBS_DIR pela chave textual eixo/grupo/marco/tarefa. Renomear um desses campos numa árvore sem atualizar a outra faz a tarefa "sumir" da comparação em vez de aparecer como divergência. O botão "Exportar PDF" do modal usa window.print() puro — depende do diálogo de impressão do navegador, sem geração programática de PDF.
